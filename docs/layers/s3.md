# S3 Gateway

You will run the S3-compatible gateway and know exactly what protocol
surface it covers.

```sh
databox gateway s3 --cluster db-node1:8443 --listen :9000
```

On the `make kind-up` cluster the gateway already runs: `make
port-forward-s3` (localhost:9000) or NodePort 30900 directly (`make
relay-s3` for sustained transfers).

Authentication uses **Databox API keys** (`databox user access-keys <name>
[--scope /prefix ...]`). A scoped key is capped to its prefixes no matter
whose key it is; the owner's grants still apply on top. In Kubernetes the
gateway resolves keys via an operator identity (`--operator-user/-password`,
env `DATABOX_OPERATOR_*`).

## S3-specific flags

| Flag | Default | Meaning |
|------|---------|---------|
| `--operator-user`, `--operator-password` | `root`, — | databox identity the gateway resolves access keys and grants as (`DATABOX_OPERATOR_USER`/`_PASSWORD`) |
| `--root-prefix` | `/s3/` | KV prefix buckets map under |
| `--multipart-ttl` | `168h` | expire unfinished multipart uploads after this long |
| `--clock-skew` | `15m` | max tolerated `x-amz-date` drift for SigV4 |

- **Auth**: AWS SigV4 with per-user access keys, both header auth and
  presigned (query-string) auth. Request timestamps outside the clock-skew
  window are rejected, and the signed payload hash is verified against the
  actual body. Mint a key pair (secret shown exactly once):

  ```sh
  databox user access-keys sam
  ```

- **Storage mapping**: bucket `b`, object `o` → key `/s3/<b>/<o>` (root
  prefix configurable). Objects are databox blobs; buckets are prefixes;
  grants apply as everywhere else (`allow list,read,write on /s3/backups` =
  bucket-scoped credentials).

- **Client config** (any standard SDK):

  ```sh
  aws s3 --endpoint-url https://localhost:9000 ls s3://backups/
  ```

- **No AWS CLI at hand?** `databox utils s3` is a minimal built-in S3
  client (SigV4) for exercising the gateway: `ls`, `mb`, `rb`, `cp`
  (with `-` for stdin/stdout), `rm`, `stat`. `--endpoint` defaults to the
  in-cluster gateway service (`http://databox-gateway-s3:9000`);
  credentials come from `--access-key`/`--secret-key` or the
  `DATABOX_S3_*`/`AWS_*` env vars:

  ```sh
  databox utils s3 mb s3://backups
  databox utils s3 cp dump.tar.gz s3://backups/dump.tar.gz
  databox utils s3 ls s3://backups --recursive
  ```

## Operation coverage

| Supported | Out of scope for v1 |
|-----------|---------------------|
| Bucket create / list / delete | Versioning |
| Object PUT / GET / DELETE / HEAD | Object lock |
| ListObjectsV2 with delimiter/CommonPrefixes + continuation tokens | ACL / policy APIs (grants are the policy system) |
| Multipart upload: initiate, UploadPart, complete, abort, ListParts | Presigned URLs beyond GET/PUT |
| Presigned GET/PUT (`X-Amz-Expires` up to 7 days) | |

Multipart parts stream into the blob engine's native chunking as temporary
blobs; complete concatenates them, in order, into the final object blob
(part boundaries need not align with chunk boundaries). Abort removes the
parts immediately; abandoned uploads older than `--multipart-ttl` are swept
lazily, so orphaned parts never accumulate.
